# はぐくみプラス AI ワークスペース

## このワークスペースについて

株式会社はぐくみプラスの業務を Claude Code で効率化するためのワークスペースです。
cocone をはじめとする D2C 通販ブランドの運営に最適化されています。
セキュリティは **3層防御**（サンドボックス・権限制御・hookスクリプト）で守られています。

---

## はぐくみプラスについて

- D2C通販メーカー（健康食品・ヘアケア・スキンケア等）
- 主要チャネル: 自社EC（定期通販）、Amazon、卸
- 主要顧客: 30-40代女性

| 事業部 | ブランド | サイト | 主な商品 |

| hugkumi+ 事業部 | cocone | https://www.hugkumiplus.net/ | クレイクリームシャンプー、ボディウォッシュ等 |
| ESIENCE 事業部 | ESIENCE | https://esience.shop/ | スキンケア・美容商品 |

### ブランドトーン

- **cocone**:「こころに、叶う」— 温かみ・丁寧さ・共感を大切にする
- **ESIENCE**: サイト（https://esience.shop/）のトーンに沿う

## 業務ルール

- **薬機法・景表法に注意**: 化粧品・健康食品の広告表現は法令遵守。NG表現を含む出力は必ず確認を求めること
- **ブランドトーン**: 各ブランドのトーンに沿ったコンテンツを作成する。事業部によって異なるので確認すること
- **顧客データの取り扱い**: 個人情報（名前・住所・メール等）は匿名化・マスキングしてから処理する
- **Notion操作**: 共有DBへの書き込みは必ず確認を取ること。全文検索は使わない
- **Slack操作**: 送信前に必ず文面を確認。全メッセージ検索は使わない

## 原則

1. **ドキュメントファースト** — 行動する前に書く。goal.md が揃うまで実装に入らない
2. **振り返りなき完了なし** — タスク完了時は必ず /task-review で振り返る
3. **学びを蓄積する** — 繰り返し使える改善は Memory に残す
4. **改善を提案する** — パターンを見つけたら Skill 化・CLAUDE.md 更新を提案する

## ワークフロー・オーケストレーション

### 1. 計画してから動く（Plan Mode Default）

- 3ステップ以上の作業は **必ず先に計画を立てる**
- 途中でおかしくなったら **即停止して再計画**
- 計画は検証ステップも含めて立てる
- 詳細な仕様を先に書いて曖昧さを減らす

### 2. サブエージェント戦略

- メインの会話を汚さないために、調査・分析・並行作業は **別のエージェントに委任** する
- 1タスク → 1サブエージェント（チームで動かすイメージ）
- 複雑な問題にはサブエージェントを増やしてコンピュートを投入する

### 3. 自己改善ループ

- ユーザーに指摘されたら、そのパターンを学びとして記録する
- 同じミスを二度しない仕組みを自分で作る

### 4. 完成前に必ず検証する

- 「できた」と言う前に本当に動くか自分で証明する
- 自問する: **「これをスタッフエンジニアが見たら承認するか？」**

### 5. エレガントさを求める（ただし過剰設計はNG）

- ハック的な修正はしない
- 「今の知識で一番きれいな解決策は何か」を常に考える
- ただし過剰設計もNG — ちょうどよさを求める

## タスク管理

1. **Plan First**: `tasks.md` にチェック可能な計画を書く
2. **Verify Plan**: 実装前にユーザーに確認する
3. **Track Progress**: 進捗に合わせて完了マークをつける
4. **Explain Changes**: 各ステップで何をしたかを簡潔に説明する
5. **Document Results**: 結果を `tasks.md` のレビューセクションに記録する
6. **Capture Lessons**: ユーザーから指摘があったら学びを記録する

## タスク構造

タスクは `tasks/{YYYY-MM-DD}_{slug}/` に格納する。

```
tasks/{YYYY-MM-DD}_{slug}/
├── README.md    … 概要・背景・リンク
├── goal.md      … 目的・ゴール・理想・要件
├── tasks.md     … TODO・進捗
├── review.md    … 振り返り・学び
├── input/       … 入力データ・参考資料
└── output/      … 成果物
```

---

## 3つの根本原則

- **Simplicity First** — 変更はできるだけシンプルに。インパクトを最小限に
- **No Laziness** — 根本原因を見つける。一時的な誤魔化しはしない。プロの基準で
- **Minimal Impact** — 必要な部分だけ触る。余計なものを巻き込まない

## セキュリティ方針

### 機密情報の保護

- `.env`、秘密鍵（`.pem`, `.key`）、認証情報を読み取らない
- パスワード、APIキー、トークンをログや出力に含めない

### 破壊的操作の防止

- 広範囲なファイル削除は実行しない
- `eval`, `exec` による動的コマンド実行は行わない

### ネットワーク通信の制御

- `curl`, `wget` による外部通信は行わない（`WebFetch` ツールを使用する）

### データ保護

- 大きなファイル（1MB超）をそのまま読み込まない — 必要な部分だけ抽出する
- 個人情報は匿名化・マスキングしてから処理する

## MCP 操作ルール

### Slack
- **NG**: キーワードで全メッセージ検索
- **OK**: チャンネル指定で直近メッセージ取得 / スレッドURL指定で取得
- **送信前**: 必ず文面をユーザーに提示して確認

### Notion
- **NG**: キーワード全文検索
- **OK**: ページ URL / DB URL を受け取って操作する
- **書き込み前**: 変更内容を必ずユーザーに確認してから実行
- 共有 DB への書き込みは特に慎重に

---

## Git操作ルール

- コミットメッセージは**日本語**で書く
- `main`/`master` ブランチへの直接pushは禁止
- `git push --force` は禁止
- `git add .` / `git add -A` は禁止
- コミット前に `git diff` で変更内容を確認する

---

## ファイル操作の方針

- 既存ファイルを編集する前に必ず内容を確認する
- 新規ファイル作成は必要最小限にする
- `.claude/` 配下の設定ファイルやスクリプトは変更しない

## スキルの利用

`.claude/skills/` 配下に以下のスキルが利用可能です:

| コマンド | 用途 |
|---------|------|
| `/secretary` | パーソナル秘書（エントリーポイント。TODO管理、壁打ち、メモ） |
| `/company` | 仮想会社組織（秘書が窓口、部署に振り分け） |
| `/task-start` | タスク開始（ドキュメント→人材配置→合意→TODO生成） |
| `/task-review` | タスク振り返り（評価→学び→改善提案） |
| `/skill-create` | 新しいスキルを対話的に作成・改善 |
| `/marketing-strategy` | MKT1メソッドによるマーケティング戦略構築（D2C対応） |
