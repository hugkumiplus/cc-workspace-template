# CC Workspace Template

非エンジニアでも安全に Claude Code を業務利用できるワークスペーステンプレートです。

---

## 設計思想

### 「AIスタッフを雇うように、AIを育てる」

このテンプレートは、Claude Code を**チャットAI**ではなく**AIスタッフ**として扱うために設計されています。

人間のスタッフを雇ったとき、最初にやることは何でしょうか？

1. **業務マニュアルを渡す** → `CLAUDE.md`
2. **作業場所を決める** → フォルダ構成（`tasks/`, `memory/`）
3. **使えるツールを教える** → MCP連携（Notion, Slack）
4. **やっていいこと・ダメなことを伝える** → セキュリティ設定
5. **繰り返し業務を仕組み化する** → スキル（`.claude/skills/`）
6. **経験から学ばせる** → Memory（`memory/`）、振り返り（`/task-review`）

このテンプレートは、これらすべてが最初から揃った状態でスタートできます。

### 3つの原則

| 原則 | 意味 |
|------|------|
| **ドキュメントファースト** | 行動する前に書く。goal.md が揃うまで実装に入らない |
| **振り返りなき完了なし** | タスク完了時は必ず振り返り、学びを蓄積する |
| **計画してから動く** | 3ステップ以上の作業は必ず先に計画を立てる |

### 仮想組織モデル

```
あなた → 秘書（窓口） → CEO（振り分け） → 各部署
```

- **秘書**: 常に窓口。何でも相談OK。TODO管理、壁打ち、メモ
- **CEO**: 裏方で判断。部署が必要な案件を自動振り分け
- **各部署**: 専門領域のタスクを担当

ユーザーは部署を意識する必要なし。秘書に話しかけるだけ。

## なぜこのテンプレートが必要か

Claude Code をそのまま使うと、こうなりがちです：

| よくある問題 | このテンプレートの解決策 |
|------------|---------------------|
| 毎回同じ説明をする | `CLAUDE.md` で業務マニュアルを自動読み込み |
| ファイルがどこに保存されるか分からない | `tasks/` フォルダ構成で作業場所を固定 |
| AIが勝手に危険な操作をする | 3層防御（サンドボックス・権限制御・hookスクリプト） |
| Notion/Slackで意図しない操作をする | MCP操作ルール（全文検索禁止、リンク渡し、書き込み前確認） |
| 繰り返し業務を毎回説明する | スキルでコマンド化（`/task-start` 等） |
| 前回の経験が次に活きない | Memory + `/task-review` で学びを蓄積 |
| やることが曖昧なまま作業が始まる | `/task-start` で goal.md 合意→TODO生成 |

## セットアップ

### 前提条件

- [Claude Code](https://claude.ai/code) がインストール済み
- Git がインストール済み（`git --version` で確認）

### Step 1: クローン

```bash
git clone https://github.com/your-company/cc-workspace-template.git
cd cc-workspace-template
```

### Step 2: Claude Code を起動

```bash
claude
```

セキュリティ設定とスキルが自動で適用されます。

### Step 3: 秘書を起動

```
/secretary
```

対話的にあなた専用の管理環境がセットアップされます。

## 使えるスキル

スキルとは、Claude Code に事前に読み込ませておく「指示書」です。
`/コマンド名` と打つだけで、定型業務を実行できます。

### ワークフロー系

| コマンド | 説明 |
|---------|------|
| `/secretary` | パーソナル秘書。TODO管理、メモ、壁打ち、ダッシュボード |
| `/company` | 仮想会社組織。秘書→CEO→部署の流れで業務を管理 |
| `/task-start` | タスク開始。ドキュメント作成→人材配置→合意→TODO生成 |
| `/task-review` | タスク振り返り。成果評価→学び抽出→改善提案 |

### ツール系

| コマンド | 説明 |
|---------|------|
| `/skill-create` | 自分だけのカスタムスキルを対話的に作成 |
| `/marketing-strategy` | MKT1メソッドによるマーケティング戦略構築（7つのエクササイズ） |

### スキルの2つの使い方

1. **手順書型**（How）: 「こうやって作業してね」— `/task-start`, `/skill-create`
2. **知識保持型**（What）: 「これを知っておいてね」— `/marketing-strategy`

知識保持型スキルは、一度構築すると以降のすべての会話で自動参照されます。
マーケティング戦略スキルのように、戦略の中身そのものをスキルに格納しておくと、
すべてのアウトプットが戦略に沿ったものになります。

**ワークフロー系** — タスク実行のフロー
- `/secretary`: パーソナル秘書。毎日のタスク・メモ・相談窓口
- `/company`: 仮想組織。大規模案件をCEOが部署に振り分け
- `/task-start`: タスク開始。goal.mdが揃うまでドキュメント作成
- `/task-review`: 振り返り。成果評価→学びをMemoryに記録

**ツール系** — 知識や成果物を作成
- `/skill-create`: 繰り返し業務をスキル化
- `/slide-create`: 12プリセットからプレゼンテーション自動生成
- `/marketing-strategy`: MKT1フレームワークで7ステップ戦略構築

## タスク管理

タスクは `tasks/{YYYY-MM-DD}_{slug}/` に格納されます。

```
tasks/2026-03-29_lp-renewal/
├── README.md    … 概要・背景・リンク
├── goal.md      … 目的・ゴール・理想・要件
├── tasks.md     … TODO・進捗
├── review.md    … 振り返り・学び
├── input/       … 入力データ・参考資料
└── output/      … 成果物
```

### PDCAサイクル

```
/task-start → 作業 → /task-review
     ↑                     │
     │              Memory に学び蓄積
     │                     │
     └─────────────────────┘
       次の /task-start に反映
```

## セキュリティ

### 3層防御

| 層 | 仕組み | 役割 |
|---|--------|------|
| 第1層 | サンドボックス | OS レベルでファイル・ネットワークを制限 |
| 第2層 | 権限制御（deny/allow） | 危険なコマンド・機密ファイルをブロック |
| 第3層 | hook スクリプト | 実行前後でパターン検査・プロンプトインジェクション検出 |

### プロンプトインジェクション対策

外部データ（Webページ、Slackメッセージ等）に含まれる悪意ある指示からAIを守ります。

| 攻撃パターン | 対策 |
|------------|------|
| リモートコード実行（`curl \| bash`） | 実行前にブロック |
| Base64難読化 | 実行前にブロック |
| 機密情報の外部送信 | 実行前にブロック |
| 外部データ内の悪意ある指示 | 実行後に警告 |
| MCP全文検索による情報漏洩 | search系ツールをdeny |

### MCP操作ルール

| やること | やらないこと |
|---------|------------|
| ページURLを渡して操作 | 「全部検索して」と言わない |
| 特定チャンネルのみ読む | 全チャンネルを開放しない |
| 書き込み前に必ず確認 | 自動投稿・自動更新を許可しない |

詳細は [docs/SECURITY_POLICY.md](docs/SECURITY_POLICY.md) を参照。

## CLAUDE.md の設計思想

`CLAUDE.md` はAIへの業務マニュアルです。Anthropic社内で実践されているベストプラクティスを組み込んでいます。

### ワークフロー・オーケストレーション

| 原則 | 説明 |
|------|------|
| **計画してから動く** | 3ステップ以上は必ず計画。おかしくなったら即停止・再計画 |
| **サブエージェント戦略** | 調査・分析は別エージェントに委任。メインの会話を汚さない |
| **自己改善ループ** | 指摘を受けたら記録し、同じミスを二度しない |
| **完成前に必ず検証** | 「スタッフエンジニアが承認するか？」と自問する |
| **エレガントさを求める** | ハック禁止。ただし過剰設計もNG |

### 3つの根本原則

- **Simplicity First** — 変更はできるだけシンプルに
- **No Laziness** — 根本原因を見つける。一時的な誤魔化しはしない
- **Minimal Impact** — 必要な部分だけ触る

## フォルダ構成

```
cc-workspace-template/
├── CLAUDE.md                        ← AIへの業務マニュアル（原則・ルール）
├── .claudeignore                    ← AI検索対象から除外するファイル
├── .claude/
│   ├── settings.json                ← 権限制御（deny/allow）+ hookスクリプト設定
│   ├── scripts/                     ← hookスクリプト（5本）
│   │   ├── block-dangerous-commands.sh  … 危険コマンドのブロック
│   │   ├── block-main-push.sh           … main直接pushのブロック
│   │   ├── protect-sensitive-files.sh   … 機密ファイル保護
│   │   ├── protect-data.sh              … 大容量データの警告
│   │   └── scan-tool-output.sh          … 出力のインジェクション検査
│   └── skills/                      ← スキル（/コマンド で実行）
│       ├── secretary/               … パーソナル秘書
│       ├── company/                 … 仮想会社組織
│       ├── task-start/              … タスク開始ワークフロー
│       ├── task-review/             … タスク振り返り
│       ├── skill-create/            … スキル作成ツール
│       └── marketing-strategy/      … マーケティング戦略構築
├── tasks/                           ← タスクフォルダ（作業はここに蓄積）
├── memory/                          ← 学びの記録（Memory）
└── docs/
    └── SECURITY_POLICY.md           ← セキュリティポリシー詳細
```
