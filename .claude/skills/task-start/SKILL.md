# タスク開始ワークフロー

## 概要

ドキュメントファーストの原則に従い、実装前にゴール・要件を明確にし、Company の最適な人材配置のもとで作業を開始する。

## ワークフロー

### Step 1: タスク情報のヒアリング（Interactive）

`AskUserQuestion` でタスクの概要を聞く。

> 新しいタスクを始めますね。以下を教えてください:
> 1. **何をやるか**（概要）
> 2. **なぜやるか**（目的・背景）
> 3. **関連リンクや資料**（あれば）

slug名もユーザーと相談して決める。日付は今日の日付を使う。

### タスクフォルダの命名規則
- 形式: `tasks/{YYYY-MM-DD}_{slug}/`
- slug: **英数字とハイフンのみ**。日本語は使わない
- 例: `tasks/2026-03-30_lp-renewal/`、`tasks/2026-04-01_sns-campaign/`

### Step 2: フォルダ・ドキュメント作成（Automatic）

`tasks/{YYYY-MM-DD}_{slug}/` に以下を自動生成:

1. **README.md** — ヒアリング内容から概要・背景・リンクを記載
2. **goal.md** — 以下の構造で作成:

```markdown
## 目的（WHY）
{ヒアリングから記入}

## ゴール（完了条件）
{ユーザーと対話して明確化}

## 理想像
{制約がなければどうあるべきか}

## 要件
{具体的な仕様・制約}
```

3. **tasks.md** — 空テンプレート:

```markdown
## TODO

## メモ
```

4. **review.md** — 空テンプレート:

```markdown
## 振り返り
（タスク完了時に /task-review で記入）

## 良かった点

## 改善点

## 学び
```

5. **input/** と **output/** ディレクトリ

### Step 3: goal.md の対話的な深掘り（Interactive）

goal.md の各セクションについてユーザーと対話し、内容を充実させる。
特に「ゴール（完了条件）」は曖昧にせず、検証可能な形にする。

> goal.md のドラフトを作成しました。確認してください:
> [goal.md の内容を表示]
>
> 修正や追加はありますか？

### Step 4: Company CEOによる人材配置（Automatic）

`.company/` が存在する場合、CEOの振り分けロジックに従って最適な部署をアサインする。

1. `.company/CLAUDE.md` と `.company/ceo/CLAUDE.md` を読み込む
2. goal.md の内容を分析
3. CEOの振り分け基準に基づき、主担当と連携部署を決定
4. アサイン結果をユーザーに報告:

> CEOが以下の人材配置を決定しました:
> - **主担当**: {部署名} — {理由}
> - **連携**: {部署名} — {理由}
>
> この配置でよろしいですか？

5. `ceo/decisions/` にアサインログを記録

※ `.company/` が未セットアップの場合はこのステップをスキップする。

### Step 5: 合意とTODO生成（Interactive → Automatic）

1. goal.md の内容を確認し、実現可能性や懸念を報告
2. ユーザーと合意を取る
3. 合意後、goal.md の要件を分解して tasks.md にTODOを生成:

```markdown
## TODO
- [ ] {要件から分解したタスク1}
- [ ] {要件から分解したタスク2}
- [ ] テスト・検証
- [ ] /task-review で振り返り
```

> tasks.md にTODOを生成しました。作業を開始しましょう！

## 重要な注意事項

- goal.md が合意されるまで実装に入らない
- ユーザーの Memory（memory/）を確認し、過去の好みや学びを反映する
- 既存タスクとの重複がないか tasks/ を確認する
- TODO の最後には必ず「/task-review で振り返り」を入れる
- タスク完了の判断はユーザーが行う。完了と言われたら「/task-review で振り返りましょう」と自然に誘導する
